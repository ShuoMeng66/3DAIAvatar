# ElderTalk 性能与延迟基准

## 延迟目标

### 全双工对话端到端延迟目标（首字延迟 < 2s）

| 环节 | 目标耗时 | 说明 |
|------|----------|------|
| VAD 检测 | < 200ms | 浏览器端 Web Audio API 实时分析，语音活动检测 |
| ASR 识别 | < 500ms | 上传音频 → FunASR/OmniSenseVoice 返回文本 |
| LLM 首字 | < 1000ms | 流式输出第一个 token 到达时间 |
| TTS 首字 | < 300ms | 首个音频 chunk 生成时间 |
| Avatar 首帧 | < 500ms | 数字人首帧渲染时间 |
| **端到端合计** | **< 2.5s** | **用户说完到数字人开口的首个字** |

### 优化策略

#### 1. 流式 LLM 输出
- 使用 SSE (Server-Sent Events) 推送 token
- 不等待 LLM 完整输出，分段送给 TTS
- LLM 首字到达后立即开始 TTS 生成

#### 2. 数字人特征预加载
- 固定 avatar 在服务启动时一次性完成特征提取
- 避免每次对话重复提取特征（节约 ~2s）
- 实现方式：启动时加载 avatar 图片 → MuseTalk/SadTalker 预处理 → 缓存特征向量

#### 3. 音频缓存
- 常见问候语 TTS 结果预生成缓存
- 缓存命中时直接播放，跳过 LLM + TTS 环节

#### 4. 并行流水线
- VAD 结束 → 同时启动 ASR + 准备数字人渲染环境
- LLM 首字输出 → 并行 TTS + Avatar 推理

## 资源占用

| 资源 | 优化前 | 优化后 | 说明 |

| 显存 | ~13GB | ~10GB | 共享 KV cache + 特征预加载 |

| 磁盘 | ~30GB | ~20GB | 清理冗余模型权重 |

## 全双工打断延迟

| 场景 | 延迟 | 说明 |
|------|------|------|
| VAD 检测到语音 | < 100ms | 浏览器端实时分析 |
| 发送 interrupt | < 50ms | 网络请求 |
| 后端停止生成 | < 100ms | 取消 LLM/TTS/Avatar 任务 |
| 前端停止播放 | < 50ms | 停止 `<audio>` 和 `<video>` |
| **总打断延迟** | **< 300ms** | **用户开口到数字人停止说话** |

## 关键性能指标 (KPI)

- [ ] 首字延迟 < 2s（90% 分位）
- [ ] 打断延迟 < 300ms（95% 分位）
- [ ] 端到端对话周期 < 5s（含 VAD + ASR + LLM + TTS + Avatar）
- [ ] 并发对话数 ≥ 1（单 GPU）
- [ ] WebRTC 视频帧率 ≥ 25fps
- [ ] 服务可用性 ≥ 99%（健康检查通过率）
